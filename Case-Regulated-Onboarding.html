Khushboo Khan Case Study/Regulated Onboarding/2025
Product Design · 2025

Recovery
Is the System.

Onboarding doesn’t fail where users get stuck. It fails where they can’t find their way back.

Role
Product Design Lead
Scope
End-to-end onboarding system
Timeline
2025, 12 weeks to ship
Domain
Regulated KYC & Identity
Framing NEW

A system deciding what to do
when its users are uncertain.

This project is not labeled as AI work, but it operates on the same decision structures. It is identity verification in a regulated flow. But the underlying problem is a decision-systems problem: observe behaviour across a sequence, infer whether the user is in a success state or a failure state, and decide what the product should do in response.

That shape, observation to inference to intervention, is the same shape an AI-assisted product carries whenever it has to behave well under ambiguous input.

What follows is the story of a system that had behaviour for success and silence for failure, and the design work that gave it a second mode.

Opening

Activation was under target.
The brief was the wrong size.

The team framed it as a flow-length problem. The data said otherwise.

Users were not failing because onboarding was long.

They were failing because the product had nothing to say to them once they got stuck.

The fix was not shorter onboarding.
It was onboarding that assumed users would get stuck and designed for that moment.
Diagnosis

The drop-off was the symptom.
The failure was upstream.

The drop-off curve looked like a friction problem. Concentrated falls at verification and document upload, consistent with what the team expected.

The second pass on the data told a different story.

<5%
of users who dropped off had interacted with help before leaving. The support surface existed. It sat behind a small icon in the corner. Users in trouble were not ignoring support. They could not see it.
9 steps
Every one a potential exit, with no recovery path visible between them.
1 icon
A corner help affordance that behaved the same regardless of user state. Functionally invisible at the moment of friction.
0 signal
Stuck users were not tracked as a cohort. The failure mode had no metric.

The product was optimised for users who were succeeding. It had no behaviour for users who were not.

Help was present.
It just wasn’t visible.
Reframe

Progression and recovery
are different problems.

The team had been solving one and assuming the other would follow.

Progression is the work of moving a user forward.

Recovery is the work of catching one who has fallen out.

A flow can be excellent at the first and silent on the second. This one was.

Naming the difference was the highest-leverage decision in the project.
No code changed before this conversation.

It stopped the debate over step count.

It shifted compliance from gatekeeper to partner, because the question changed from “what can we remove” to “how do stuck users find their way back.”

It gave engineering and support a shared target.

Every decision after it was downstream of the new definition.

Decision system NEW

Onboarding became a system
with two modes, chosen in response to signal.

Once recovery existed, the product had a second way to behave. Which mode it was in at any moment had to be a decision, not an assumption.

The system watches a small set of signals across the verification and routes its behaviour accordingly.

Fig 03 Signal, inference, response
Progression
signal
Steady input, forward motion, clean validations
The system stays quiet. The recovery affordance steps back. Default: minimise interface weight.
Friction
signal
Dwell time, revisits, hesitation at verification steps
Recovery affordance gains weight in context. Guidance surfaces before the user exits. Default: make recovery legible.
Failure
signal
Retry loops, failed validation, repeat drop-offs
The system escalates toward human-assisted recovery. The user is handed a route, not an error. Default: keep the human in the loop.
The system does not pick the path for the user. It adjusts its own visibility in proportion to how confident it is that the user is on track, and hands control back at the point that confidence breaks down.
AI-assisted systems carry the same problem
in a different form.
What does the interface do when the model is uncertain, when the output is partially right, when the user doesn’t know what to trust? Design the failure state first.

Recovery, viewed this way, is not a support feature. It is the behaviour of a system designed to handle its own uncertainty and to return control to the human at the right moment.

Nine screens became five. No required input was removed. Four of the cuts came from the system carrying work the user had been asked to do: consent became a modal, country and document type became a read-at-capture inference, and the two capture, two preview, and separate retry routes collapsed into one capture and one preview with retry inline.
System behaviour
The country selection screen and the separate document type step were not moved. They were deleted. The system reads the document at the moment of capture and infers both country and document type directly from what the user uploads. The user was being asked to declare information the system could already derive. Removing the step removed an entire class of mismatch error between user selection and uploaded document, which had been a silent source of retries earlier in the funnel.

Compression

Fewer steps for the user. The system carries the rest.

Nine screens to five. Every regulated input preserved, but the work was moved. Consent, previously a full screen, became a modal over the first step.

Country and document type stopped being a user decision. The system now reads the document at capture and infers both directly from the upload, which made the selection screen redundant and cut a silent class of user-side mismatch errors.

Two capture screens, two preview screens, and a separate retry route collapsed into one capture for both sides and one preview with retry inline.

Compliance signed off because the inputs were still there. Users felt the speed because the transitions were not.

Shift · user-carried work → system-carried work
Visibility

Recovery became a first-class, context-aware surface.

The corner help icon was replaced by a labelled entry point that changes weight with context. The interface began responding to step-level signal instead of behaving identically everywhere. Prominent at historically high-friction steps, quiet where users were cruising.

Help now reads as part of the system, not as an escape hatch outside it.

Shift · state-agnostic surface → context-aware intervention
Instrumentation

The failure mode became a metric.

Recovery rate was introduced as a tracked metric. Of users who show a signal of being stuck, the percentage who re-enter the flow and complete.

The team did not have this number before. Once it existed, the team could tell whether recovery design was working independently of raw completion.

Shift · raw completion → recovery rate
Decisions

The trade-offs
made deliberately.

Three decisions carried the project. Each was documented in writing so the team would not re-litigate them every sprint.

Fig 04 Decision record
Restructure
over removal
Simplicity vs. compliance
The regulated inputs could not be cut. Restructuring recovered most of the speed a shorter flow would have delivered, without moving the compliance line.
Visibility
over minimal UI
Aesthetic vs. activation
A quieter interface was easier to defend on craft grounds. The activation number could not afford it. Added weight was accepted in exchange for a surface stuck users could find.
Clarity
over speed
Compression vs. comprehension
Document and liveness steps carry the most friction and the most regulatory nuance. Further compression would have saved steps and lost users. Compression stopped where clarity mattered more.
Shipped decisions get re-litigated. Documented ones do not. Writing them down was part of the design.

Principles NEW

What this system teaches
about designing under uncertainty.

The case is specific. The principles are not. They apply to any product whose input is ambiguous, whose output is sometimes wrong, or whose user can fall out of the happy path without the system noticing.

Failure-first

Design the failure state before the happy path.

If the system has no behaviour for the moment a user gets stuck, the user has no route back. This holds for a regulated onboarding. It holds for a model that returns low-confidence output. It holds for any system where the input is noisy and the path is not guaranteed.

Applies to KYC · AI-assisted flows · any decision surface
Legible confidence

Make the system’s confidence readable to the user.

A recovery affordance that does not change weight by context is telling the user nothing. The surface should get louder where the system has reason to expect friction, and quieter where it does not. The user should be able to read, at a glance, how certain the system is that they are on track.

Applies to context-aware UI · model confidence · progressive disclosure
Human-retained

Intervention should increase agency, not replace it.

The user in a failure state is the actor. The system’s job is to make the next move legible, not to pick it. Human-in-the-loop is not a fallback pattern reserved for edge cases. It is the default posture when a system is operating under uncertainty.

Applies to human-in-the-loop · assistive systems · escalation design
Transfer

Where this thinking transfers
beyond onboarding.

The system described here sits in identity verification. The structure of the problem does not.

Any product that operates under uncertainty carries the same shape:

AI systems

When model output is partial, uncertain, or wrong

The system must decide when to intervene, how to communicate confidence, and when to return control to the user. Recovery is not an edge case. It is the main path.

Marketplaces

When supply, demand, or trust breaks down

Users fall out of flows not because the path is unclear, but because the system does not respond to breakdowns. Recovery systems restore momentum.

Enterprise tools

When workflows fail under real operational pressure

Operators need systems that detect friction, surface guidance, and maintain accuracy under load. Silence is failure.

This is not onboarding design.
It is designing how systems behave when they are no longer certain.
In hindsight

In onboarding, progress matters.
Recovery determines whether users finish.
